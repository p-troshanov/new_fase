# План следующих улучшений

Всё без подключения реальных платёжных провайдеров, рассрочек и внешних аналитик — только продуктовая часть на текущем стеке (TanStack Start + Lovable Cloud).

---

## 1. Авторизация и роли

- Включить email + Google авторизацию через `/auth` (страница уже есть — довести до рабочей).
- Создать таблицу `profiles` (id → auth.users, full_name, phone, avatar_url) + автосоздание при регистрации.
- Роли через отдельную таблицу `user_roles` (`app_role` enum: `admin`, `student`) и security-definer функция `has_role()`. Никогда не хранить роль в profiles.
- Пометить текущего заказчика админом сид-миграцией по email.

## 2. Личный кабинет `/account`

Защищённый layout `src/routes/_authenticated/`:

- `/account` — профиль (имя, телефон, аватар), редактирование.
- `/account/orders` — список заказов пользователя со статусами (`pending`, `paid`, `failed`, `refunded`), способом оплаты, ссылкой на детали.
- `/account/orders/$id` — карточка заказа: состав, сумма, статус, история, кнопка «повторить оплату» для `pending/failed`.
- `/account/materials` — доступ к купленным курсам/интенсиву (пока просто заглушка с материалом по `entitlement`).
- Привязать существующие заказы к `user_id` (миграция: добавить колонку + backfill по email).

## 3. Админка `/admin`

Гейт по роли `admin` (редирект на главную, если нет роли):

- `/admin/orders` — таблица всех заказов: фильтры по статусу/датам/способу, поиск по email, экспорт в CSV.
- `/admin/orders/$id` — ручная смена статуса, отметка «возврат», кнопка «переотправить доступ».
- `/admin/leads` — таблица из `leads` (уже есть), пометка «обработан», заметки.
- `/admin/funnel` — простая сводка по `funnel_events`: посещения → клик оплатить → создан заказ → оплачен (без внешних систем, чисто из БД).

## 4. Лиды и уведомления

- Форма «Оставить заявку» на лендинге (имя, телефон/email, интерес: курс/интенсив) → server fn пишет в `leads`.
- Отправка уведомления админу на email при новой заявке и при успешной оплате (через Resend — попросить у пользователя API-ключ, когда дойдём).
- Пользователю: письмо-подтверждение заявки и письмо с доступом после `paid` (шаблоны — заглушки текста).

## 5. Контент и SEO

- Отдельные посадочные с уникальным `head()` (title, description, og:title/description/url, canonical):
  - `/course`, `/intensive` — уже есть, дописать уникальные мета.
  - `/about`, `/faq`, `/reviews`, `/contacts` — новые страницы.
- Блоки: FAQ со `schema.org/FAQPage`, отзывы со `schema.org/Review`, программа курса.
- `public/sitemap.xml` со всеми публичными маршрутами, `robots.txt` без `Disallow: /`.
- Единый компонент `SeoHead` не нужен — используем `head()` в каждом route-файле.

## 6. Мелкое качество

- Страница `/legal/privacy` и `/legal/offer` (текст-заглушки под редактирование).
- Единый компонент `EmptyState` и `ErrorState` для админки/кабинета.
- Тост-уведомления на ключевых действиях (сохранение профиля, смена статуса заказа).

---

## Что НЕ входит в этот этап

- Реальный CloudPayments widget и HMAC (остаётся в текущем виде-заглушке).
- Долями / Халва / OZON / WB рассрочки (UI-блок уже скрыт).
- Внешняя аналитика (Метрика/GA4) и пиксели.

## Порядок выполнения (по шагам, с проверкой)

1. Миграции: `profiles`, `user_roles`, `has_role()`, `orders.user_id`.
2. Авторизация + `/auth` + Google + гейт `_authenticated`.
3. Личный кабинет (профиль → заказы → материалы).
4. Админка (заказы → лиды → воронка).
5. Форма лидов + email-уведомления (после того как дадите Resend-ключ).
6. Контент-страницы + SEO + sitemap.
7. Юридические страницы + полировка UX.
